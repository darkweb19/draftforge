through `doctor` without exposing secret values, and by wiring the role-routed

configured adapter in addition to the existing manual prompt/submit flow. Add

## Acceptance criteria

- `doctor` reports, for each configured role adapter, whether the harness command
  is on PATH or the API key variable is set, plus an authentication status, and
  never prints a key, token, or other secret value.
- Absent authentication is reported as `missing`, not `fail`, so a
  partially-configured machine still passes; only genuine misconfiguration is
  `fail`.
- `plan` can drive one architect turn through the configured adapter using the
  existing `runArchitectTurn`, and the manual `--prompt`/`--submit` path keeps
  working unchanged.
- Authentication and capability discovery perform no action that mutates the
  project or leaks a secret.
- Contract tests pass for every adapter; live smoke tests pass for each locally
  available auth mode and are skipped, not failed, when that mode is absent.
- `.draftforge/state.json` marks `phase-03` complete only after this exit gate,
  and `SESSION.md` is regenerated.
- `npm run check` passes.

## Verification

Run `npm run check`, then `draftforge doctor`. Run available live smoke tests for
any authenticated harness or API key present in the environment.
