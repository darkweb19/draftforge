## Acceptance criteria

- The architect response envelope has a strict runtime contract in
  `src/domain/architect.ts`; it is either a question batch or a plan.
- Raw model text is parsed in `src/application/`, tolerating a fenced code block
  around the JSON, and is validated before it reaches domain state.
- `buildArchitectPrompt` is deterministic: identical inputs produce byte-identical
  prompts, and the prompt carries the source draft, repository rules, the current
  planning revision, and already-answered questions.
- The prompt requests one complete question batch, never a follow-up drip.
- `src/application/ports.ts` declares the model-runner port only; no adapter, no
  vendor SDK, and no import from `src/providers/`.
- `applyArchitectResponse` is pure and reuses `submitQuestionBatch` and
  `submitPlan`; it never bypasses their guards.
- `draftforge plan --prompt` prints the prompt for the current planning stage.
- `draftforge plan --submit <file.json>` applies a recorded architect response
  under the project lock and rejects a stage mismatch.
- `draftforge plan --answer <id>=<text>` records interview answers and can be
  repeated; unknown IDs and empty answers are rejected.
- An approved plan cannot be replaced by `--submit`; the error names plan
  revision as the required path.
- Deterministic fixtures drive the full CLI loop: start, prompt, submit
  questions, answer, prompt, submit plan, approve.
- No provider call, adapter, or authentication is added.
- `npm run check` passes.
