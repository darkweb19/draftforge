## Acceptance criteria

- Architect question and plan responses have strict runtime contracts.
- The architect submits at most one complete question batch per planning revision.
- Answers persist in `.draftforge/planning.json` and can resume without repeating
  answered questions.
- Plans reject duplicate IDs, unknown or self dependencies, and dependency cycles.
- No implementation task is runnable before explicit plan approval.
- Approval requires all blocking questions to be answered and a valid plan; root
  tasks in the active phase become ready only after approval.
- Approval materializes phase, ADR, and task files before publishing runnable
  state and can be retried without resetting task progress.
- Project-relative planning paths cannot escape the project boundary.
- Interrupted writers leave a recoverable project lock.
- Deterministic fixtures cover fresh, resumed, invalid, and approved planning flows.
- Provider calls and live provider authentication remain outside this task.
- `npm run check` passes.
