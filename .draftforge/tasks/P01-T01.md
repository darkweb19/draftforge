# P01-T01 — Implement local project initialization

Status: done

## Objective

Implement `draftforge init [directory]` so it creates a resumable, provider-neutral project without making a model call.

## Owned paths

- `src/commands/init.ts`
- `src/config/`
- `src/state/`
- `templates/`
- Relevant tests and documentation.

## Dependencies

- P00-T01

## Acceptance criteria

- Refuses to overwrite non-DraftForge files without explicit approval.
- Creates valid canonical state, `SESSION.md`, `AGENTS.md`, `CLAUDE.md`, phase files, and task directories.
- Re-running is idempotent or reports an actionable conflict.
- Works on Windows, macOS, and Linux path semantics.
- No provider, CLI login, or API key is required.

## Evidence

- `src/commands/init.ts` implements the command; `src/config/config.ts`, `src/state/initial-state.ts`, `src/state/templates.ts`, and atomic writes in `src/state/files.ts` support it. Project files are rendered from `templates/`.
- Conflict rule: missing file → create; identical content → leave; anything else → report and write nothing, unless `--force`. An initialized project is only gap-filled, never rewritten.
- `test/init.test.ts` covers fresh init, directory-name defaulting, idempotent re-run, restoring a deleted file, conflict refusal, `--force`, untouched unrelated files, and corrupted state. `test/templates.test.ts` guards schema drift between `templates/schema/` and `.draftforge/schema/`.
- Verified with `npm run typecheck`, `npm run lint`, `npm test` (16 passing), `npm run build`, plus a built-CLI smoke run in a temporary directory covering init, re-init, `status`, `handoff`, and a refused conflict (exit 1).
- Paths are built with `node:path`; the smoke run was executed on Windows.
